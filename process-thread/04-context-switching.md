# 04. 컨텍스트 스위칭 비용 — CPU 캐시와 TLB 플러시

## 🎯 핵심 질문

- 컨텍스트 스위칭 시 커널은 실제로 무엇을 저장하고 복원하는가?
- TLB 플러시가 성능에 미치는 영향은 얼마나 큰가?
- Tomcat 스레드 풀과 Netty 이벤트 루프의 컨텍스트 스위칭 비용 차이는?
- 컨텍스트 스위칭이 많다는 것을 어떻게 감지하는가?

---

## 🔍 왜 이 개념이 실무에서 중요한가

**`top`을 봤더니 CPU 사용률은 30%인데 서버 응답이 느리다.** `us`(유저)는 낮고 `sy`(시스템)가 높다면 컨텍스트 스위칭이 과도하게 발생하는 것이다. CPU가 실제 작업보다 스위칭에 더 많은 시간을 쓰고 있다.

**스레드 풀을 500으로 늘렸더니 오히려 처리량이 줄었다.** CPU 코어가 16개인 서버에서 스레드 500개가 실행되면 500/16 = 약 31개의 스레드가 코어 하나를 경쟁한다. 컨텍스트 스위칭 오버헤드가 실제 작업 시간보다 커질 수 있다.

**Redis가 단일 스레드임에도 빠른 이유 중 하나가 컨텍스트 스위칭 비용 제거다.** 이벤트 루프 모델은 스레드 간 스위칭이 없어 CPU 캐시를 최대한 활용한다.

## 🔬 내부 동작 원리

### 컨텍스트 스위칭의 전체 과정

컨텍스트 스위칭은 현재 실행 중인 태스크를 멈추고 다른 태스크로 CPU를 넘기는 과정이다.

```
[태스크 A 실행 중]
       │
       │ 1. 스위칭 트리거 발생:
       │    - 타임슬라이스 소진 (타이머 인터럽트)
       │    - I/O 대기 (블로킹 시스템 콜)
       │    - 높은 우선순위 태스크 등장
       │
       ▼ [커널 진입 (인터럽트/예외)]
       │
       │ 2. 현재 CPU 레지스터 상태 저장 (태스크 A의 PCB에)
       │    - 범용 레지스터: rax, rbx, rcx, rdx, rsi, rdi, r8~r15
       │    - 명령어 포인터: rip (다음 실행할 명령어 주소)
       │    - 스택 포인터: rsp
       │    - 세그먼트 레지스터, EFLAGS
       │
       │ 3. 스케줄러: 다음 실행할 태스크 B 선택 (CFS)
       │
       │ 4. 메모리 컨텍스트 전환
       │    - 태스크 A와 B가 다른 프로세스라면:
       │      → CR3 레지스터 변경 (페이지 테이블 루트 포인터)
       │      → TLB 플러시 (!)
       │    - 같은 프로세스의 스레드라면:
       │      → CR3 변경 없음 (페이지 테이블 공유)
       │      → TLB 플러시 없음 (!)
       │
       │ 5. 태스크 B의 CPU 레지스터 복원 (B의 PCB에서)
       │
       ▼ [태스크 B 실행 재개]
```

### TLB 플러시가 성능에 미치는 영향

TLB(Translation Lookaside Buffer)는 가상 주소 → 물리 주소 변환을 캐싱하는 CPU 내부 캐시다.

```
TLB 히트 (변환 결과 캐싱됨):
  가상 주소 → TLB 조회 → 물리 주소
  비용: ~1 사이클

TLB 미스 (캐싱 없음):
  가상 주소 → TLB 미스 → 페이지 테이블 워크 → 물리 주소
  비용: ~수십~수백 사이클 (메모리 접근 필요)

프로세스 스위칭 후 TLB 플러시:
  ┌─────────────────────────────────────────┐
  │ TLB가 비워짐 (다른 프로세스의 매핑 무효화)       │
  └─────────────────────────────────────────┘
  새 프로세스 실행 초기: TLB 미스 폭발
  → 메모리 접근마다 페이지 테이블 워크 필요
  → 성능 저하 (워밍업 구간)

스레드 스위칭 시: TLB 플러시 없음
  → 같은 프로세스 = 같은 페이지 테이블
  → TLB 항목 유지
  → 스위칭 비용이 프로세스보다 훨씬 낮음
```

### CPU 캐시 오염 (Cache Thrashing)

컨텍스트 스위칭 시 CPU L1/L2/L3 캐시도 문제가 된다.

```
태스크 A 실행 중:
  L1 캐시: A의 핫 데이터로 채워짐 (캐시 히트율 높음)

컨텍스트 스위칭 → 태스크 B 실행:
  L1 캐시: B의 데이터로 교체됨 (A의 데이터 밀려남)
  초기 L1 캐시 미스 발생 → L2/L3 또는 메모리 접근

다시 태스크 A로 스위칭:
  L1 캐시: 다시 A의 데이터 재적재 필요
  → 캐시 워밍업 비용 반복

스위칭이 잦을수록:
  → 캐시 히트율 감소
  → 메모리 대역폭 소비 증가
  → 실제 작업 처리량 감소
```

```
프로세스 스위칭 비용 (예시, 현대 CPU):
  레지스터 저장/복원:       ~1 μs
  TLB 플러시 + 재로딩:    ~10~100 μs (캐시 워밍업 포함)
  L1/L2 캐시 워밍업:      ~수십 μs

스레드 스위칭 비용 (같은 프로세스):
  레지스터 저장/복원:       ~1 μs
  TLB 플러시 없음:         0
  L1 캐시: 부분적 재사용   ~수 μs
```

### Tomcat vs Netty 컨텍스트 스위칭 비교

```
Tomcat (BIO/NIO 스레드 모델):
  1000개 동시 연결 = 200개 스레드 처리 + 800개 대기

  타임슬라이스마다:
  [스레드 1] → [스레드 2] → ... → [스레드 200]
  → 초당 수천~수만 회 컨텍스트 스위칭

Netty (이벤트 루프):
  1000개 동시 연결 = 이벤트 루프 스레드 8개 (CPU 코어 수)

  이벤트가 없으면 epoll_wait()에서 대기 (스위칭 없음)
  이벤트 발생 시만 처리 → 스위칭 최소화

  초당 컨텍스트 스위칭: 수백 회 수준
```

### 실험 1: 컨텍스트 스위칭 현황 모니터링

```bash
# vmstat으로 초당 컨텍스트 스위칭 수 확인

# cs 컬럼: 초당 컨텍스트 스위칭 수
# 참고 기준:
#   < 1,000/s: 정상
#   1,000~10,000/s: 주의 (애플리케이션 특성에 따라 다름)
#   > 50,000/s: 과도, 스레드 수 줄이기 고려

$ pidstat -w 1 5
# cswch/s: 자발적 스위칭/초 (I/O 대기)
# nvcswch/s: 비자발적 스위칭/초 (CPU 경합)

# nvcswch/s가 높은 프로세스 = CPU 과부하 상태
```

### 실험 2: 스레드 수와 처리량 관계 실험

## 📊 성능/비용 비교

| 스위칭 유형 | TLB 플러시 | 캐시 영향 | 비용 (대략) |
|------------|-----------|---------|-----------|
| 프로세스 간 스위칭 | O (전체 플러시) | 높음 (전체 캐시 교체) | ~10~100 μs |
| 같은 프로세스 스레드 간 | X (없음) | 중간 (일부 공유) | ~1~10 μs |
| 같은 코어에서 VM 종류 | O + 추가 | 높음 | ~100 μs+ |

```

100개      4ms           ~25,000/s         ~5-10%
500개      4ms           ~125,000/s        ~20-40%
1000개     4ms           ~250,000/s        ~40-80% ← 심각
```

## ⚖️ 트레이드오프

**적은 스레드 (이벤트 루프 모델)**:
- 컨텍스트 스위칭 최소화, 캐시 효율 극대화
- 단점: 블로킹 코드가 하나라도 있으면 전체 이벤트 루프 멈춤
- 적합: Redis 같은 인메모리 작업, Nginx의 HTTP 처리

**많은 스레드 (스레드 풀 모델)**:
- 블로킹 I/O 작업(DB 쿼리, 외부 API 호출)에서 자연스럽게 동작
- 단점: 컨텍스트 스위칭 비용, 메모리 소비
- 적합: Spring MVC + JDBC(블로킹 DB 드라이버)

**ASID(Address Space ID)로 TLB 플러시 최적화**:

최신 CPU는 ASID를 지원해 프로세스 스위칭 시 TLB 전체를 플러시하지 않고 ASID로 구분한다. 하지만 ASID 슬롯이 제한적이어서 프로세스 수가 많으면 여전히 플러시가 발생한다.

## 📌 핵심 정리

```
컨텍스트 스위칭 = 레지스터 저장/복원 + (프로세스 간이면) TLB 플러시

비용 순서:
  프로세스 스위칭 > 스레드 스위칭 > (이상적으로) 스위칭 없음

TLB 플러시가 비싼 이유:
  - TLB 비워짐 → 첫 메모리 접근마다 페이지 테이블 워크
  - CPU L1/L2 캐시도 교체됨

감지 방법:
  vmstat 1 → cs 컬럼 (초당 컨텍스트 스위칭)
  top → sy 높음 = 스위칭 오버헤드
  pidstat -w → 프로세스별 자발적/비자발적 스위칭
  /proc/<pid>/status → voluntary/nonvoluntary_ctxt_switches

설계 원칙:
  CPU 코어 수보다 스레드를 훨씬 많이 만들면 역효과
  I/O 바운드: 스레드 수 ≈ CPU 코어 * (1 + I/O대기/CPU사용)
  CPU 바운드: 스레드 수 ≈ CPU 코어 수
  이벤트 루프: 스위칭 최소화 → 캐시 효율 극대화
```

## 🤔 생각해볼 문제

**Q1.** CPU 8코어 서버에서 스레드 8개만 실행 중일 때 컨텍스트 스위칭이 발생할 수 있는가?

<details>
<summary>해설 보기</summary>

**그렇다.** 커널 스레드(kworker, ksoftirqd 등)도 컨텍스트 스위칭을 유발한다. 또한 I/O 대기 중인 스레드가 실행 가능 상태로 돌아오면 스케줄러가 개입한다. 그러나 유저 스레드만 8코어 8스레드라면 스케줄링 경합이 거의 없어 컨텍스트 스위칭이 최소화된다.

</details>

**Q2.** 같은 프로세스의 두 스레드 간 스위칭이 다른 프로세스 간 스위칭보다 왜 빠른가?

<details>
<summary>해설 보기</summary>

같은 프로세스의 스레드는 동일한 페이지 테이블(mm_struct)을 공유한다. 따라서 스위칭 시 CR3 레지스터 변경이 없고 TLB 플러시가 불필요하다. 또한 L1/L2 캐시에 이미 올라온 데이터를 공유할 수 있어 캐시 워밍업 비용도 줄어든다. 반면 다른 프로세스로 스위칭 시에는 완전히 다른 주소 공간으로 전환되므로 TLB와 캐시가 무효화된다.

</details>

**Q3.** `vmstat`에서 `cs`(컨텍스트 스위칭)가 50,000/s로 높다. 이것만으로 문제가 있다고 판단해야 하는가?

<details>
<summary>해설 보기</summary>

**단독으로는 판단하기 어렵다.** 컨텍스트 스위칭 자체보다 `sy` CPU 비율을 함께 봐야 한다. `sy`가 30% 이상이라면 스위칭 오버헤드가 실제 작업에 영향을 주는 것이다. 또한 `nvcswch/s`(비자발적 스위칭)가 높으면 CPU 경합이 심한 것이고, `cswch/s`(자발적 스위칭)가 높으면 I/O 대기가 많은 것으로 해석이 다르다. 애플리케이션 특성(I/O 바운드 vs CPU 바운드)과 함께 해석해야 한다.
